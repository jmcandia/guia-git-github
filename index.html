01 · Fundamentos

¿Qué es Git y para qué sirve?

Imagina que estás escribiendo un documento de Word y quisieras poder volver a cualquier versión anterior, ver quién cambió qué y cuándo, y trabajar en equipo sin pisarse los cambios. Eso es exactamente lo que hace Git con tu código.

Git es un sistema de control de versiones distribuido. En palabras simples: es una herramienta que guarda el historial completo de todos los cambios que haces en tu código. Si en algún momento algo deja de funcionar, puedes volver atrás. Si alguien del equipo rompe algo, puedes ver exactamente qué cambió.

💡 Dato clave

Git fue creado en 2005 por Linus Torvalds (el mismo creador de Linux) para gestionar el desarrollo del kernel de Linux. Hoy, prácticamente toda la industria del software lo usa.

🕐

Historial completo

Cada cambio queda registrado. Puedes viajar al pasado y recuperar cualquier versión de tu proyecto.

👥

Trabajo en equipo

Varias personas pueden trabajar al mismo tiempo en el mismo proyecto sin conflictos (o resolviéndolos cuando ocurren).

🌿

Ramas paralelas

Puedes desarrollar nuevas funcionalidades de forma aislada, sin tocar el código que ya funciona.

🌐

Distribuido

Cada desarrollador tiene una copia completa del repositorio. No hay un único punto de falla.

Git vs GitHub — ¿son lo mismo?

No, son cosas distintas pero complementarias:

analogía 
# Git  →  es como Microsoft Word (el programa)
# GitHub →  es como OneDrive (el lugar donde guardas tus documentos en la nube)

Git    = la herramienta que corre en tu computador
GitHub = la plataforma web donde subes y compartes tus repositorios

Puedes usar Git sin GitHub (guardando todo de forma local), pero usar GitHub sin Git no tiene sentido. En la práctica, siempre van de la mano.

✅ Alternativas a GitHub

Existen otras plataformas similares como GitLab y Bitbucket. Todas funcionan con Git, así que si aprendes a usar Git + GitHub, puedes adaptarte fácilmente a cualquiera de ellas.

02 · Configuración inicial

Instalación y configuración

Antes de empezar, necesitas tener Git instalado en tu computador y configurar tu identidad.

Instalar Git

Windows

Descarga el instalador desde git-scm.com. Durante la instalación, deja las opciones por defecto. Se instala también Git Bash, una terminal que te permite ejecutar comandos Git.

macOS

terminal · macOS 
# Opción 1: Con Homebrew (recomendado)
brew install git

# Opción 2: Instalando Xcode Command Line Tools
xcode-select --install

Linux (Ubuntu/Debian)

terminal · Linux 
sudo apt update && sudo apt install git

Verificar la instalación

terminal 
git --version
# git version 2.44.0

Configuración inicial (obligatoria)

Lo primero que debes hacer es decirle a Git quién eres. Esta información aparece en cada commit que hagas:

terminal 
# Tu nombre (el que aparecerá en el historial)
git config --global user.name "Tu Nombre"

# Tu correo (el mismo que usas en GitHub)
git config --global user.email "tucorreo@ejemplo.com"

# Editor por defecto (VSCode en este caso)
git config --global core.editor "code --wait"

# Verificar que quedó todo bien
git config --list
⚠️ Importante

El flag --global aplica la configuración para todos los repositorios en tu computador. Si quieres una configuración distinta para un proyecto específico, usa el mismo comando sin --global dentro de ese repositorio.

03 · Conceptos clave

Los conceptos que necesitas entender

Antes de aprender los comandos, hay que entender el vocabulario. Son pocos conceptos, pero son fundamentales.

El repositorio (repo)

Un repositorio es la carpeta de tu proyecto pero con superpoderes: contiene todos tus archivos y el historial completo de cambios. Cuando haces git init, Git crea una carpeta oculta llamada .git que guarda todo ese historial.

Las tres zonas de Git

Este es el concepto más importante. En Git, tus archivos pueden estar en tres estados distintos:

las tres zonas 
Working Directory   →   Staging Area   →   Repository
  (tu carpeta)           (zona de espera)        (historial)

       git add ────────────▶         git commit ────────────▶
📁

Working Directory

Tu carpeta de trabajo normal. Aquí editas y creas archivos. Los cambios todavía no están "guardados" en Git.

📋

Staging Area (Index)

Una zona de preparación. Aquí pones los cambios que quieres incluir en el próximo commit. Es como armar una caja antes de enviarla.

📦

Repository (.git)

El historial permanente. Una vez haces commit, los cambios quedan registrados para siempre en el historial local.

¿Qué es un commit?

Un commit es una "foto" de tu proyecto en un momento específico. Cada commit tiene:

  • Un identificador único (hash SHA-1), algo como a3f2c91
  • El autor (nombre + email)
  • La fecha y hora
  • Un mensaje descriptivo
  • Los cambios en los archivos

Las ramas (branches)

Una rama es una línea paralela de desarrollo. La rama principal se llama main (o master en proyectos más viejos). Puedes crear ramas para desarrollar funcionalidades nuevas sin afectar el código que ya funciona.

📌 Analogía

Imagina que estás escribiendo una novela. La rama main es tu manuscrito oficial. Si quieres probar un final alternativo, creas una rama nueva, escribes el final alternativo, y si te gusta, lo fusionas con el manuscrito principal. Si no te convence, simplemente eliminas esa rama.

04 · Comandos esenciales

Los comandos que usarás todos los días

No necesitas memorizar todos los comandos de Git. Con estos 15 ya puedes trabajar de forma profesional.

Crear e inicializar repositorios

terminal 
# Crear un repositorio nuevo en la carpeta actual
git init

# Clonar un repositorio existente desde GitHub
git clone https://github.com/usuario/repositorio.git

# Clonar en una carpeta con nombre específico
git clone https://github.com/usuario/repo.git mi-carpeta

El ciclo básico: add → commit

flujo de trabajo diario 
# Ver el estado actual de tus archivos
git status

# Agregar UN archivo al staging area
git add archivo.py

# Agregar TODOS los archivos modificados
git add .

# Crear un commit con un mensaje
git commit -m "feat: agrega autenticación de usuarios"

# Ver el historial de commits
git log

# Historial resumido (más legible)
git log --oneline --graph --all

Trabajar con ramas

branches 
# Ver todas las ramas (la activa tiene un *)
git branch

# Crear una nueva rama
git branch feature/login

# Cambiar a una rama existente
git checkout feature/login

# Crear una rama Y cambiarse a ella (atajo)
git checkout -b feature/login

# Forma moderna (Git 2.23+)
git switch -c feature/login

# Fusionar una rama hacia la rama actual
git merge feature/login

# Eliminar una rama (solo si ya fue mergeada)
git branch -d feature/login

Sincronizar con GitHub (remote)

remote · sincronización 
# Subir cambios al repositorio remoto
git push origin main

# Bajar cambios del repositorio remoto
git pull origin main

# Ver los remotos configurados
git remote -v

# Agregar un remoto (cuando usas git init en vez de clone)
git remote add origin https://github.com/usuario/repo.git

Deshacer errores

deshacer cambios 
# Sacar un archivo del staging (sin perder los cambios)
git restore --staged archivo.py

# Descartar los cambios de un archivo (¡cuidado! no hay vuelta atrás)
git restore archivo.py

# Deshacer el último commit (mantiene los cambios en staging)
git reset --soft HEAD~1

# Ver diferencias entre archivos
git diff

# Guardar cambios temporalmente sin hacer commit
git stash
git stash pop   # recuperar lo guardado
⚠️ Con cuidado

git reset --hard es el comando más peligroso de Git. Elimina los cambios de forma permanente. Úsalo solo si estás seguro de lo que haces.

05 · GitHub

Trabajando con GitHub

GitHub es la red social del código. Es donde subes tus repositorios, colaboras con otros y compartes tu trabajo con el mundo.

Flujo básico de trabajo

1

Crear un repositorio en GitHub

Ve a github.com → botón "New" → ponle nombre, elige si es público o privado y crea el repo.

2

Conectar tu repo local con GitHub

Si ya tienes código local, conectas tu repositorio con el remoto usando git remote add origin URL.

3

Subir tu código (push)

Con git push origin main subes tus commits al repositorio en GitHub. La primera vez puede que necesites autenticarte.

4

Pull Requests (PR)

Cuando trabajas en equipo, no fusionas directamente. Creas un Pull Request para que otro miembro revise tu código antes de aprobarlo.

5

Issues y Proyectos

GitHub tiene un sistema de tareas (Issues) y tableros de proyecto para organizar el trabajo del equipo.

El archivo .gitignore

Es un archivo especial donde le dices a Git qué archivos NO debe rastrear. Útil para excluir claves, dependencias y archivos del sistema:

.gitignore · Python + Node 
# Dependencias de Node
node_modules/

# Variables de entorno (¡NUNCA subas tus claves!)
.env
.env.local

# Python
__pycache__/
*.pyc
venv/
.venv/

# Java / Maven
target/
*.class

# IDEs
.idea/
.vscode/
*.suo
✅ Tip

Puedes generar automáticamente un .gitignore para tu lenguaje en gitignore.io.

06 · Conventional Commits

Escribir commits que tengan sentido

¿Has visto repositorios con mensajes de commit como "fix", "cambios", "arreglo final final 2"? Eso hace imposible entender el historial. Conventional Commits es el estándar de la industria para escribir mensajes que sí tienen sentido.

🔗 Recurso oficial

La especificación completa está en conventionalcommits.org — también disponible en español.

La estructura de un Conventional Commit

formato 
tipo(alcance): descripción corta

[cuerpo opcional: explica el QUÉ y el POR QUÉ]

[footer opcional: BREAKING CHANGE, closes #123]

Tipos de commits

feat
Nueva funcionalidad Agrega algo nuevo al proyecto. Equivale a un incremento menor de versión (1.0 → 1.1).
fix
Corrección de bug Arregla un error existente. Equivale a un patch de versión (1.0.0 → 1.0.1).
docs
Documentación Cambios solo en documentación: README, comentarios en el código, wikis.
style
Formato / estilo Cambios que no afectan la lógica: espacios, punto y coma, indentación, formateo.
refactor
Refactorización Reestructurar código sin cambiar su comportamiento. No es un fix ni una feat.
test
Tests Agregar o modificar pruebas unitarias o de integración.
chore
Tareas de mantenimiento Actualizar dependencias, configurar herramientas, tareas que no tocan el código de producción.
perf
Performance Cambios que mejoran el rendimiento de la aplicación.
ci
Integración continua Cambios en archivos de CI/CD: GitHub Actions, pipelines, scripts de deploy.

Ejemplos reales

commits · ejemplos 
# Feat simple
feat: agrega formulario de registro de usuarios

# Feat con alcance
feat(auth): implementa login con Google OAuth

# Fix con referencia a issue
fix(api): corrige error 500 al crear usuario duplicado

Closes #42

# Breaking change (cambio que rompe compatibilidad)
feat(api)!: cambia endpoint de /users a /api/v2/users

BREAKING CHANGE: el endpoint anterior queda obsoleto.
Actualizar todos los clientes que usen /users.

# Refactor con contexto
refactor(services): extrae lógica de email a EmailService

Se separa la lógica de envío de emails del controlador
para mejorar la separación de responsabilidades (SRP).

# Docs
docs: actualiza README con instrucciones de instalación

¿Por qué usarlo?

📖

Historial legible

Cualquiera puede entender qué pasó en el proyecto con solo leer el historial de commits.

🤖

Automatización

Herramientas como semantic-release pueden generar versiones y changelogs automáticamente basándose en tus commits.

🔍

Facilita revisión

En los Pull Requests, es mucho más fácil revisar código cuando los commits tienen mensajes descriptivos.

✅ Consejo

No es obligatorio usar Gitmoji. Lo importante es que tu equipo defina un estándar y lo aplique de forma consistente. Si el equipo acuerda usar solo Conventional Commits sin emojis, está perfecto. Lo que no funciona es que cada persona escriba los commits como quiera.

08 · Estrategias de ramificación

Branching Strategies

Una estrategia de branching define cómo un equipo organiza sus ramas en Git. No hay una única respuesta correcta — la mejor estrategia depende del tipo de proyecto y del equipo. Veamos las tres más populares.

Git Flow

Creado por Vincent Driessen en 2010. Es el modelo más completo y estructurado. Define cinco tipos de ramas con roles bien definidos. Fue diseñado para proyectos con releases planificadas y ciclos de versión largos.

Las cinco ramas de Git Flow

main
v1.0
v1.1
v2.0
develop
feature/*
release/*
hotfix/*

main (o master)

Siempre contiene el código en producción. Solo recibe merges desde release/* y hotfix/*. Cada merge en main se etiqueta con una versión (v1.0, v2.0, etc.).

develop

La rama de integración. Aquí se juntan todas las features terminadas. Es la fuente de verdad del trabajo en curso.

feature/*

Una rama por cada nueva funcionalidad. Sale de develop y vuelve a develop cuando está lista. Ej: feature/login-social

release/*

Se crea cuando develop está listo para salir a producción. Se hacen los últimos ajustes, se sube la versión y luego se mergea a main y a develop.

hotfix/*

Para arreglos urgentes en producción. Sale directamente de main y se mergea a main y develop. Ej: hotfix/fix-login-crash

Ejemplo de flujo completo

git flow · ejemplo 
# 1. Iniciar una feature desde develop
git checkout develop
git checkout -b feature/sistema-pagos

# 2. Trabajar en la feature (varios commits)
git add . && git commit -m "feat(pagos): agrega integración con Transbank"
git add . && git commit -m "feat(pagos): implementa flujo de reembolso"

# 3. Merge de feature a develop
git checkout develop
git merge --no-ff feature/sistema-pagos
git branch -d feature/sistema-pagos

# 4. Crear rama release cuando todo está listo
git checkout -b release/1.2.0
# ... ajustes finales, bump de versión ...
git commit -m "chore: bump version to 1.2.0"

# 5. Mergear a main y a develop
git checkout main
git merge --no-ff release/1.2.0
git tag -a v1.2.0 -m "Release v1.2.0"
git checkout develop
git merge --no-ff release/1.2.0
📌 Úsalo cuando...

Tu proyecto tiene releases formales con versiones (v1.0, v2.0), hay un QA que debe validar antes de subir a producción, o el equipo es mediano/grande y necesita mucha estructura. Ejemplos: software de escritorio, librerías, apps móviles, sistemas bancarios.

GitHub Flow

Creado por GitHub en 2011 como una alternativa más simple a Git Flow. Solo usa dos tipos de ramas: main y ramas de feature. La idea central es que main siempre está lista para ser desplegada a producción.

main
deploy
deploy
deploy
deploy
feature/A
PR ✓
feature/B
PR ✓

Las reglas de GitHub Flow

1

main siempre está deployable

El código en main siempre debe poder subirse a producción en cualquier momento. Si no puede, hay un problema.

2

Crea una rama descriptiva para cada tarea

Cada feature, fix o mejora va en su propia rama con nombre descriptivo: feature/agregar-buscador, fix/corregir-paginacion.

3

Haz commits frecuentes y sube (push) a menudo

Commitea con frecuencia y sube a GitHub aunque no esté terminado. Sirve como respaldo y permite feedback temprano.

4

Abre un Pull Request para pedir revisión

Cuando la feature está lista, abres un PR hacia main. Aquí el equipo revisa, comenta y aprueba el código.

5

Mergea y despliega

Una vez aprobado el PR, se hace merge a main y se despliega a producción. Idealmente de forma automática (CI/CD).

github flow · ejemplo 
# 1. Crear rama desde main
git checkout main
git pull origin main
git checkout -b feature/agregar-buscador

# 2. Desarrollar y commitear
git commit -m "feat(search): agrega barra de búsqueda en el header"
git commit -m "feat(search): implementa búsqueda por nombre y RUT"

# 3. Subir la rama a GitHub
git push origin feature/agregar-buscador

# 4. Abrir Pull Request desde GitHub.com
# → Compare & pull request → Descripción → Assign reviewers

# 5. Una vez aprobado: merge a main (desde GitHub o terminal)
git checkout main
git merge feature/agregar-buscador
git push origin main
# → CI/CD hace el deploy automáticamente 🚀
📌 Úsalo cuando...

Tu equipo hace deploy continuo (varias veces al día o a la semana), el proyecto es una aplicación web/API donde siempre hay una sola versión en producción, o el equipo es pequeño y necesita simplicidad. Ejemplos: startups, SaaS, proyectos web, APIs REST.

Trunk-Based Development

Es el modelo más radical y el más usado en empresas como Google, Facebook y Amazon. La idea es simple pero requiere disciplina: todos los desarrolladores trabajan directamente en la rama principal (el "trunk" o main), haciendo commits pequeños y frecuentes.

main (trunk)
ramas cortas
(opcionales)
< 2 días

Principios clave

⚡

Commits muy pequeños y frecuentes

En vez de trabajar días en una feature y hacer un commit grande, se hacen commits pequeños varias veces al día.

🚩

Feature Flags

Para subir código de funcionalidades incompletas sin que los usuarios las vean. El código está en producción pero "apagado".

🤖

Tests automatizados

Sin una buena cobertura de tests, esto no funciona. Los tests automatizados son el escudo de seguridad que permite commitear con confianza.

🔄

Integración continua (CI)

Cada push activa una pipeline que ejecuta los tests automáticamente. Si algo falla, el equipo lo sabe al instante.

trunk-based · ejemplo con feature flag 
# El código se integra a main aunque la feature no esté lista
# Un feature flag controla si el usuario la ve o no

# Ejemplo en Python (feature flag simple)
FEATURE_FLAGS = {
    "nuevo_dashboard": False,  # apagado en producción
    "exportar_excel": True,   # activo para todos
}

def mostrar_dashboard(user):
    if FEATURE_FLAGS["nuevo_dashboard"]:
        return nuevo_dashboard(user)
    return dashboard_clasico(user)

# El equipo integra el código, lo activa cuando está listo
# sin necesidad de hacer merge de una rama gigante
📌 Úsalo cuando...

El equipo tiene alta disciplina técnica, buena cobertura de tests, CI/CD maduro, y el objetivo es maximizar la velocidad de entrega. Equipos de alto rendimiento en empresas como Google o Meta lo usan. No recomendado para equipos nuevos o proyectos sin tests.

Comparación de estrategias

Criterio Git Flow GitHub Flow Trunk-Based
Complejidad Alta (5 tipos de ramas) Baja (2 tipos) Mínima (1 rama)
Velocidad de entrega Media (releases planeadas) Alta (continua) Muy alta (varias/día)
Control de calidad Muy alto (QA formal) Alto (code review) Depende de tests
Versiones múltiples ✓ Soporta múltiples ✗ No ideal ✗ No ideal
Hotfixes de emergencia ✓ Rama hotfix formal Rama rápida a main ✓ Directamente a trunk
Requiere CI/CD Recomendado Sí, importante Obligatorio
Requiere tests automatizados Recomendado Recomendado Obligatorio
Ideal para Apps móviles, librerías, software empaquetado SaaS, APIs, web apps Equipos de alto rendimiento
Popularidad actual Decayendo Muy popular En aumento

¿Cómo elegir la estrategia correcta?

Aquí hay un árbol de decisión para ayudarte a elegir:

¿Tu proyecto maneja múltiples versiones en producción al mismo tiempo?

(ej: v1.x para clientes antiguos, v2.x para clientes nuevos)

→ Sí Git Flow
→ No → Continúa abajo
¿Tu equipo tiene procesos de QA formales o ciclos de release largos?

(ej: releases mensuales, aprobaciones formales, auditorías)

→ Sí Git Flow
→ No → Continúa abajo
¿Tu equipo tiene buena cobertura de tests y CI/CD maduro?

(tests unitarios, de integración, pipeline automatizada)

→ Sí, muy buena Trunk-Based
→ No o recién empezando GitHub Flow
✅ Recomendación para proyectos de curso

Para proyectos de curso o equipos que recién empiezan, GitHub Flow es la opción ideal. Es simple, se aprende rápido, incentiva el uso de Pull Requests para revisión de código, y refleja bien el flujo de trabajo de la industria para la mayoría de los proyectos web.
